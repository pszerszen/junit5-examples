JUnit 5 - first gaze

February 24, 2017, Piotr Szerszeń

Testing your applications is often a part of development just as important as programming. Of course testing everything manually would be awfully time consuming and tiring.
If you thing "automated testing" - you thing JUnit. Some time ago my attention has been caught by the very new, the 5th, version of this framework. I decided to share a few facts and code samples.

First thing to mention is that JUnit 5 is technically speaking a very new framework rather than next version of the one we already know. This is actually a good news. The previous version is decent, but it still has a few serious flaws which couldn't be got rid of with simple fixes. We needed a new approach in some cases - now we can have it. Most of equivalent annotations has different name and of of them are in another package. This is letting to use JUnit 3/4 and JUnit 5 in parallel. So any project currently using JUnit 4 can start using JUnit 5 and developers can start writing new tests with new framework. Legacy tests can be (wel they should be) migrated gradually.

SET UP:

To use both JUnits in maven project you need to add these dependencies in your pom.xml

<dependency> <groupId>org.junit.jupiter</groupId> <artifactId>junit-jupiter-api</artifactId> <version>LATEST</version> <scope>test</scope> </dependency> <dependency> <groupId>org.junit.jupiter</groupId> <artifactId>junit-jupiter-engine</artifactId> <version>LATEST</version> <scope>test</scope> </dependency> <dependency> <groupId>org.junit.vintage</groupId> <artifactId>junit-vintage-engine</artifactId> <version>LATEST</version> </dependency> <dependency> <groupId>junit</groupId> <artifactId>junit</artifactId> <version>LATEST</version> <scope>test</scope> </dependency> <dependency> <groupId>org.assertj</groupId> <artifactId>assertj-core</artifactId> <version>LATEST</version> <scope>test</scope> </dependency> <dependency> <groupId>org.mockito</groupId> <artifactId>mockito-all</artifactId> <version>LATEST</version> <scope>test</scope> </dependency>

Big data and machine learning are topics rapidly growing in popularity. Big data’s purpose is to collect massive amounts of domain-specific data, containing lots of useful information. For example, we can use all the customer data of a retail shop to find out what a retailer can do to increase sales and maximize income.

Machine learning is ideal for exploiting the potential of big data. It is a set of specific algorithms which use training data to create a model describing every piece of information as a set of features. Based on that model, you can use other data with a similar structure but incomplete features to infer the missing information.

Let’s consider customer preferences: we can provide our algorithm’s data regarding our current customers’ shopping preferences (which items they were looking for and what they bought) and based on that information, when a new customer arrives looking for some item, our machine learning model can provide suggestions as to what customer may also like to buy.

Our team decided to explore these opportunities by developing its own system based on such concepts. We came up with the idea of teaching our model which opinions related to a given topic were positive and which were negative. Choosing a topic with which to work is up to the user. Using this system, when new opinions arrived our model was able to classify them into previously defined categories. With this information, we could provide statistics on how opinions changed over time so we could, for example, see how some information about a given topic affected people’s opinions.

We developed a microservices-based application with the given stack:

  • Java 8
  • Apache Spark (Spark SQL, Spark Mlib)
  • Apache Cassandra
  • Apache ActiveMQ (communication between services)
  • JHipster (basic web app with UI layer out of the box)
  • Spring Framework (Boot, MVC, Security)
  • Datastax Spark-Cassandra connector
  • Crawler4J
  • Twitter4J
  • Javascript
  • AngularJS

It contains four modules:

    1. Analyzer – we’ve used JHipster to kickstart our development. The Yeoman generator speeds up development. It can create a basic CRUD web application based on Spring Boot from questions which are asked during the project generation process. It provides REST endpoints, a basic AngularJS-based UI (which is easy to extend), security configuration, repositories, services and much more. It can also provide a lot of useful integration code/frameworks out of the box. For example; authentication (3 types), database (3 types), caching, search engine, WebSockets, Maven/Gradle, Sass, Angular Translate, and testing frameworks.
    1. It serves as the presentation layer for our application. We customized the generated code to our needs and extended the AngularJS front-end.
    1. Crawler – the purpose of this one is to provide us with data from web pages and social media (Twitter). We used Crawler4J and Twitter4J. Crawler4J is an open source multi-threaded web crawler for Java. Twitter4J was used for crawling Twitter API and Twitter Streaming API. These modules convert all crawled data to entities, used in our application, and store it in a database. We provided a few parsers for various web pages which we had been crawling during our development process. We used jsoup to parse HTML from crawled pages.
    1. Processor – Here we are using Apache Spark API or, more precisely, two of its modules:
      • Spark SQL – for filtering our data using search criteria defined by the user.
      • Spark Mlib – for machine learning jobs, we are using APIs for algorithms included in this module to perform model creation and the estimation of yet to be classified documents.

Here we have implemented four algorithms from the Apache Spark Mlib:

      • Linear Regression without validation.
      • Cross Validation.
      • Train Validation.
      • Multilayer Perceptron (an algorithm based on neural networks).

It also is connected to Apache ActiveMQ, which allows other modules to ask for jobs that are being performed and the return of their results.

  1. Algorithm comparator – tests algorithms implemented in the processor for accuracy and correctness. It uses Apache ActiveMQ to request results for all algorithms (one at the time) and then compares the algorithms’ accuracy. We used it to decouple the logic of algorithm comparison from the logic of algorithm execution. It also provides REST API for algorithms results visualization (charts).


We came up with following Cassandra schema:


Database Schema
 

  • Search profiles – profiles for filtering data predefined by the user.
  • Search criteria – for a given profile, we can define multiple criteria which serve as “and” relationships.
  • Documents – our data crawled/imported from the official repository lands here.
  • Results – after filtering, using a given search profile, all documents which match our predefined criteria have their representation in this table. The training process and classification of validation data are also performed in this table.
  • Classification groups – a table for algorithm comparison, only the comparison module uses this one, the rest of the modules are not aware of it. We store the classification of given result performed by all of our algorithms here.
  • Algorithm accuracy – we store the estimated accuracy for all algorithms here.

Probably many of you noticed that we’re using NoSQL database with a relational schema. We have a reason for it. Relations are used for our presentation layer. Having them in one database with data for Spark calculations makes our life easier when developing web part of our application and doing experiments for the next part of this article. (Please, keep in mind, that if our solution would be implemented on production, database infrastructure should be split into a relational and non-relational part.)

We also have a pretty standard schema for handling users, authentication, etc.

Basic learning flow for algorithms:


Basic learning flow for algorithms

Raw text is split into words, the words are then converted into feature vectors which contain more information than just words, such as the count of occurrences. Our learning algorithms use these vectors with another data classification, represented as a floating point number, to create the model for our training data. When the model is finished it is ready to predict classifications for newly arrived data. It provides the prediction of classification as a number and our code’s job is to provide an interpretation for it by converting the number to a classification.

The source code for this project is hosted on our GitHub repository.

Summary

We developed a completely functional system which is capable of gathering data, preparing it for training as well as estimating a model using prepared data. We’re still playing around with it, checking what we can gain using real data. Curious about the results? Stay tuned! The second part of our article will appear soon.
